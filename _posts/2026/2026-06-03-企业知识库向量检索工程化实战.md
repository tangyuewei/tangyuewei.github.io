# 企业知识库 + 向量检索的工程化落地实战

## 背景

去年帮一个客户做内部知识库问答系统，核心需求很直接：他们有大量 PDF 年报、Word 公告、Excel 研报，散落在各个地方，员工每次找资料都要翻半天，问问题也没人能随时回答。

老板的期望是："能不能像跟人聊天一样，直接问问题，系统就从这些文档里找到答案喂回来？"

这就是典型的 RAG（检索增强生成）场景。听起来不复杂，但真正做起来，工程上的细节比想象中多得多。

这篇文章记录一些实际踩过的坑，不是教程，更像是项目复盘。

---

## 整体思路

大方向其实就三步：

1. **把文档向量化** —— 用 Embedding 模型把文本转成向量，存进支持向量检索的数据库
2. **检索最相关的片段** —— 用户提问也转成向量，算余弦相似度，把最相关的几段文本找出来
3. **丢给 LLM 生成答案** —— 把检索到的片段作为上下文，让模型基于这些材料回答问题，减少幻觉

技术栈用的是 Spring Boot + PostgreSQL（带 PGvector 插件）+ 多家 Embedding/LLM 接口。

听起来很清晰，但真正动手的时候，第一个问题就来了。

---

## 多模型 Embedding，向量维度对不上

最开始只接了 OpenAI 的 `text-embedding-ada-002`，输出是 1536 维向量，数据库里 `question_vector` 字段定义为 `PGvector(1536)`，一切正常。

后来客户要求同时支持百度千帆和阿里 DashScope（国内访问稳定，而且有免费额度），接进去之后发现：

| 模型 | 向量维度 |
|---|---|
| OpenAI text-embedding-ada-002 | 1536 |
| 百度 bge-large-zh | 1024 |
| 阿里 text-embedding-v1 | 1024 |

数据库字段是 `PGvector(1536)`，百度/阿里返回的是 1024 维，**直接存就报错**：

```
ERROR: differently-dimensioned arrays not supported
```

当时的解法比较粗暴，写了一个 `VectorResizer`，把 1024 维的向量补零到 1536 维：

```java
// VectorResizer.java
public static float[] resizeVector(float[] originalVector, int targetSize) {
    float[] resizedVector = new float[targetSize];
    System.arraycopy(originalVector, 0, resizedVector, 0,
        Math.min(originalVector.length, targetSize));
    // 剩余维度补 0
    Arrays.fill(resizedVector, originalVector.length, targetSize, 0);
    return resizedVector;
}
```

这么做能跑，但补零的那部分维度对语义计算完全没有贡献，相似度计算会有精度损失。后来反思，更合理的做法应该是：

- **方案 A**：统一用同一个 embedding 模型，从根源上避免维度不一致
- **方案 B**：升级 PGvector 到支持动态维度的版本，不同模型存不同的字段或表

当时选了最快能上线的方案，技术债留到了现在。如果有读者面对同样的问题，建议直接走方案 A 或 B，补零只是权宜之计。

## 文本分块，比想象中重要

文档向量化之前，必须先切分成合适大小的片段。最开始我们图省事，按固定字数（比如每段 2000 字）硬切，结果很快发现检索质量很差。

原因很明显：硬切经常把一句话拦腰截断，或者把一个完整的表格切成两半，向量化之后的语义就不完整了，检索召回自然不准。

后来参考 LangChain 的思路，自己撸了一个 `RecursiveCharacterTextSplitter`，核心思想是：**优先在语义边界处分割**，而不是在固定字数处硬切。

分隔符按优先级排：

```java
// 分隔符优先级：双换行 > 单换行 > 中文句号 > 特殊标点
private List<String> separators = Arrays.asList(
    "\n\n", "\n", "。", "[﹔﹖﹗．！？]"
);
```

具体逻辑是：先尝试用 `\n\n`（段落边界）分割，如果切出来的块还是太大，就降级用 `\n`，再不行就用 `。`，以此类推。这样可以最大程度保留语义完整性。

另外有一个 `chunkOverlap` 参数（我们设的是 200 字符），让相邻块之间保留一些重叠内容，避免关键信息刚好落在块的边界上被割裂。

调用起来大概是这样：

```java
RecursiveCharacterTextSplitter splitter =
    new RecursiveCharacterTextSplitter(null, 4000, 200);
List<String> textBlockList = splitter.splitText(content);
```

4000 字符的块大小是我们实测下来比较合适的值，太小了上下文不够，太大了向量表示的语义会模糊。这个值可以根据具体业务调整。

## 向量存进 PGvector 之后，怎么查

PostgreSQL 装了 PGvector 插件之后，可以用 `<=>` 算子算余弦距离：

```java
String tmpSql =
    "SELECT pid, title, doc_source, content, answer, " +
    "1 - (question_vector <=> ?) AS cosineValue " +
    "FROM t_qa_vector " +
    "WHERE answer IS NOT NULL " +
    "  AND question_vector IS NOT NULL " +
    "ORDER BY cosineValue DESC " +
    "LIMIT ? OFFSET ?";
```

`<=>` 返回的是余弦**距离**，所以要用 `1 - 距离` 转成相似度，这样直觉上更容易理解（越接近 1 越相似）。

实际用下来有两个经验：

**1. 一定要加业务维度过滤。** 如果所有文档都存在同一张表里，不加任何过滤条件直接 `ORDER BY cosineValue DESC`，很容易召回不属于当前用户/当前业务的片段。我们按文档来源（`doc_source`）做了过滤，多租户场景下这是必须的。

**2. 分页别省。** `LIMIT` 设得太大会把大量向量数据搬回应用层，延迟明显上升。我们一般 `LIMIT 10` 就够了，毕竟送给 LLM 的上下文也不能塞太多。

## 批量生成 QA 对，线程池加速

有一个需求是：用户上传一份几万字的年报，系统自动从中生成若干组问答对（用于后续微调或者作为知识库素材）。

这份年报被分块之后可能有上百个文本块，如果每个块串行调用 LLM，耗时太长，用户体验很差。所以用线程池并发处理：

```java
List<Future<List<UserQA>>> futures = new ArrayList<>();
for (Map<String, Object> map : needGenerateQuestionListInfo) {
    Future<List<UserQA>> future = threadPoolTaskExecutor.submit(
        new GetChatGptReply(map, gptPrompt, ...)
    );
    futures.add(future);
}

// 等所有任务完成，收集结果
while (true) {
    for (Future<List<UserQA>> future : futures) {
        if (future.isDone()) {
            retList.addAll(future.get());
            futures.remove(future);
            break;
        }
    }
    if (futures.isEmpty()) break;
}
```

`GetChatGptReply` 是一个 `Callable` 实现，封装了单块文本的 QA 生成逻辑。这里有个细节：百度千帆的 embedding 接口返回的是 `List<BigDecimal>`，需要手动转成 `float[]` 再包成 `PGvector` 对象，比 OpenAI SDK 的体验差一些，但国内网络访问比较稳定，算是个取舍。

另外，用户要求生成 N 组问答，但文本块有 M 个（M 可能远大于 N），我们做了均匀采样，尽量让问答覆盖文档的各个部分，而不是全都集中在文档开头。

## 几个真实的坑

### 坑 1：百度千帆 embedding 维度不匹配（上文提过）

最开始没做 `VectorResizer`，1024 维向量直接往 `PGvector(1536)` 字段里塞，入库时报错。后来在 `questionToVector()` 和检索入口都加了 `VectorResizer.resizeVector()` 才解决。

### 坑 2：Prompt 里的正则替换 bug，藏得很深

在构造发给 LLM 的 prompt 时，有一段动态替换逻辑，意图去掉占位符 `Q2:` 和 `Q3:`：

```java
tmpGptPrompt = tmpGptPrompt.replaceAll(
    "Q2:\\s+\\nQ3:\\s+", "");
```

写出来之后一直没生效，prompt 里始终带着 `Q2:/Q3:` 残留，LLM 有时候会被搞困惑，输出的格式不对。

查了半天才发现原因：`replaceAll` 的第一个参数是**正则表达式**，`\n` 在正则里是普通字符而不是换行符，所以始终匹配不上。正确写法应该是 `\\n`，或者干脆不用 `replaceAll()`，改用 `String.replace()`（这个方法是普通字符串匹配，不是正则）。

这种 bug 不仔细看很难发现，测试用例也没覆盖到，最后是线上出了问题才回头定位到的。

### 坑 3：PGvector 写入必须用 `setObject()`

批量入库时，向量字段不能直接 `setString()`，必须用 `setObject()` 传入 `PGvector` 对象，否则 PostgreSQL JDBC 驱动报类型转换错误：

```java
// 正确
pstmt.setObject(11, qa.getQuestion_vector());  // PGvector 对象

// 错误
pstmt.setString(11, qa.getQuestion_vector().toString());  // 报错
```

这个其实在 PGvector 的文档里有写，但当时没仔细看文档，踩了坑才去查的。

## 回顾

**小亮点：**

- 多模型适配用了枚举 + 工厂模式，`CustomModel` 里封装了不同平台的调用差异，新增模型只需要加一个枚举值，改动范围可控
- 文本分块用递归分隔符策略，比固定字数切分效果好不少，这个决策回头看是对的
- PGvector 的 IVFFlat 索引在百万级数据量下检索延迟能控制在 100ms 以内，性能够用

**后续可优化：**

- 向量维度 `1536` 硬编码在好几处地方，如果要切换到 `1024` 维的模型，得改好几个文件。应该用配置文件管理目标维度
- `VectorResizer` 的补零方案是权宜之计，前面说过，应该有更优雅的做法
- Prompt 替换的正则 bug 到现在还没修（见坑 2），属于知道了但还没排期的情况
- 整个检索+生成链路都是自己手写的，没有用 LangChain4j 或者 Spring AI 这类框架，重复造了一部分轮子。如果现在重新做，会优先考虑用现有框架
